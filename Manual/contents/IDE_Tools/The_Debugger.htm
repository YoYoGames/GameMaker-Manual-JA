デバッガ

デバッガは ゲームをチェックするための強力なツールで、特にコードのエラーやバグを追跡するのに便利です。また、scripts が必要なときに実行されているか、変数や配列に何が含まれているかをチェックすることもできます。 期待する値など...GameMaker IDEからDebugger の上部にある Debug ボタン を使ってゲームを実行するか、Build メニューから"Debug" を選択すると、IDE を起動することができます。

注：Windows Firewall（または実行する他のファイアウォール・プログラム）により、このモジュールのセキュリティ例外を作成するよう促される場合があります。これを行わないと、正しく動作しません。これは Debugger が動作し、debugging モバイル・プラットフォームでは必須です。

デバッグモードでゲームを実行すると、IDE 、デバッガで利用可能なさまざまなツールが表示されます。

上の図は、debugger のデフォルトのレイアウトですが、IDE の他の部分と同様に、必要に応じてこれをカスタマイズすることができます。 不要なウィンドウを表示したり、必要なドキュメントやウィンドウの大きさを変更したりすることができます。IDE の上部にあるdebugger のコンテキストメニューから、いつでもDebugger のレイアウトをリセットしたり、閉じたウィンドウを再び開いたりすることができます。

以下では、debugger の各セクションが何のためにあるのかを説明します。

グラフ表示

グラフ表示では、プロジェクトのメモリ使用状況やフレームレートを時系列でグラフィック表示します。

グラフには、現在と平均のメモリ使用量、および現在と平均の FPS が時系列で表示され、マウスホイール とともに / を使って拡大・縮小できます。また、マウスカーソルをいずれかのグラフ要素に合わせると、その要素の選択ポイントでの値が表示されます。

グラフ表示では、特定のデバッグイベントもグラフの長さに沿って点として表示されます。これらのイベントは、ゲームの実行中に特定の重要なイベントを表示するために、GameMakerが自動的に生成します。 を使用して、独自のグラフ要素を追加することができます。 show_debug_message()と debug_event(). システムイベントは、コードによってトリガーされるイベントとは異なる色でフラグが立てられます。

デバッグでゲームを実行し、これらのイベントにマウスオーバーすると、ゲーム内のその時点で発生したすべてのイベントのリストが表示されます。 クリックすると新しいウィンドウで開きます" をクリックすると、リストを別ウィンドウで表示して修正することができます。

リソースビュー

リソースビューには、コードまたはGML Visual アクションを含むことができる、プロジェクトで使用されているすべてのリソースのリストが表示されます。このセクションは、アセットブラウザと同じように機能し、アイテムを展開することができます。 をクリックすると、より詳細な情報、たとえば、特定のobject のさまざまなイベントが表示されます。次に、任意のリソースをダブルクリック（ ）すると、そのリソースを開くことができます。 そこで値をチェックしたり、ブレークポイントを追加・削除することができます。

コールスタック

このウィンドウには、イベントの現在のコールスタックと、ステップスルーされるコードの行番号が表示されます。コールスタックは、基本的に「後入れ先出し」のデータ構造で、すべてのゲームの実行順序を表示します。 debugging は仮想マシン（VM）を使って実行されるので、GameMakerランナーからのコードと、あなたがゲーム用に書いたコードの両方が使用されます。debugger は は、ここに情報が表示される前に、一時停止して います。

ツールバー

ツールバーには、debugger のコントロールと、メモリ使用量、ゲームの現在の（実際の）FPSなどの基本情報、debugger が現在接続されているかどうかを示すインジケーターがあります（インジケーターは、接続されていると緑になります）。 のインスタンスが動作している場合は赤で表示されます)。利用可能なコントロールは以下の通りです。

一時停止したゲームを再び開始します（キーボードショートカット "f5" ）。
これでゲームが一時停止（ブレーク）します。
ゲームを再開する。
ゲームを停止させる。
コードブロックまたはscript (キーボードショートカット "f11") に足を踏み入れます。
コードブロックまたはscript (キーボードショートカット "f10") の上にステップします。
コードブロックまたはscript (キーボードショートカット + "f11") からステップアウトします。
デバッグ情報のリアルタイム更新のオン・オフを設定します（デフォルトではオフになっています）。

最初の3つのボタンは、ゲームの開始、一時停止、再開 、停止を コントロールするためのものです。ゲームが一時停止されると、（一時停止ボタン またはブレークポイントから）3つのステップボタンを使って、コードをステップ実行することができます。

コードを「ステップ」するというのは、現在のコード行を実行してから次のコード行に「ステップ」することで、ゲーム全体を一度に1行ずつ実行することができるという意味です。 ということです。通常は「ステップイン」ボタンで行いますが、「ステップオーバー 」ボタンと「ステップアウト 」ボタンも用意されています。ステップオーバーは次のような場合に便利です。 大きなscripts がある場合、これをクリックすると、script またはコードブロック全体が一度に実行され、次の行で停止します。一方、ステップアウトするとscript の外に出て（実行はされますが、一度に停止します）、次の行で停止します。 スコープが変わると、その次のステップに進みます。

最後のリアルタイム更新のアイコンは、デバッグウィンドウのデータをリアルタイムに更新することを許可するもので、変数の変化やインスタンスの作成・破棄をその場で確認できます。ただし、これはゲームのパフォーマンスに影響を与える可能性があります。 また、すべての情報がリアルタイムで利用可能なわけではありません。例えば、Render Stateウォッチ（「ウォッチ」のセクションを参照）はリアルタイムで見ることができません。

コード表示

コードビューでは、プロジェクトの実行中に、異なるscripts やイベントコードを確認することができます。 デフォルトでは空の状態で始まりますが、リソースビューから 任意のasset をダブルクリックすると、ここでそのコードを開くことができます。複数の異なるイベントやscripts を開くこともでき、それらはコードビューworkspace の上部にタブとして追加されます。

コードビューでコードウィンドウを開いているときに、ガター（行番号のある場所）で を左クリックすると、ブレークポイントを 追加したり削除したりすることができます（詳しくは breakpoints ）については、このページのさらに下を参照してください。

マウスの右ボタン（ ）でコンテキストメニューを開き、breakpoints の追加や削除、任意のコピーも可能です。 選択したコードをクリップボードにコピーします。

時計

このセクションでは、ゲームのさまざまな側面を「見る」ことで、より深く、より詳細なdebugging を可能にすることに特化しています。これについては、次のページで詳しく説明します。

  • 時計


ゲーム実行中は、デフォルトではデバッグウィンドウに変数の値やその他の詳細が更新されないので、まずDebugger の Pause ボタンを使ってゲームを一時停止してから Locals や Globals などのデータを利用するか、上部 のアイコンからリアルタイム更新を有効にする必要があることに注意します。ただし このルールの唯一の例外はProfile ビューで、これは独自のオン/オフボタンがあるため、リアルタイムswitch と独立して有効にするとリアルタイムで更新されます。このビューの詳細については、次のページを参照してください。

  • プロファイラー

debugging ゲームでもうひとつ重要なのは、ブレイクポイントの使用です。breakpoint は、プロジェクトのコードの中で、何が起こっているかを確認できるように、停止させたい場所です。Debugger を実行すると、ブレークポイントを追加したり削除したりすることができます。 を左クリックすることで、いつでも、どのコードまたはscript ブロックでも、行番号のあるガターで、またはマウスの右ボタンで をクリックします。 コンテキストメニューを開き、ブレークポイントの追加と削除を行うことができます。

ブレークポイントは、GameMaker IDE からも設定することができます。プロジェクト作成中、いつでも追加・削除できます。 f9 (または、マウスの右ボタン で、"toggle breakpoint" を選択)、object ウィンドウのアクションで行います。 (GML Visual ユーザーの場合)、またはコード・エディター・ウィンドウの行の上。これで、デバッグモードでゲームを実行すると、定義したポイント（または複数のポイント）で一時停止します。

IDE で設定したブレークポイントはプロジェクトと一緒に保存され、再度削除されるまでは様々な実行に対応します。しかし、breakpoint でコードにDebugger を追加すると、モジュールを閉じたり、新しいテスト実行を開始すると、このブレークポイントは失われます。 すべてのbreakpoints は、Output Windowに表示され、有効化、無効化、または削除することができることに注意してください。
