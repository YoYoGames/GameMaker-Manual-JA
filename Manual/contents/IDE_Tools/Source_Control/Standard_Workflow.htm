標準的なワークフロー

プロジェクトに取り組んでいるうちに、自然にsprites を作成したり、scripts を編集したり、timelines を削除したり、プロジェクトが時間とともに発展するのに必要なことは何でもするようになります。これらはすべて、何か問題が発生したときに追跡し、元に戻したいと思うような行動です。これが、Source Control を使用する主な理由です。事故があっても元に戻せるように、基本的なワークフローを最初から手早く説明し、この仕組みを理解していただきます。

最初から、新しいGML プロジェクトを作成し、メインオプションから source control を有効にします。これで上部のSource Control メニューにアクセスできるようになりましたので、早速「プロジェクトリポジトリを作成」を選択します。room のリソースを見てみると、デフォルトのroom に赤いエクスクラメーション がついています。もし、プロジェクトに他のリソースがあれば、下の画像のようにすべて同じアイコンになっているのがわかると思います。

ここで、空白のプロジェクトをそのままコミットするか、ベースとなる状態にするか、どちらかを選択することになります。とりあえず、最初のコミットを行うので、Source Control>Commit Changes をクリックします。すると、3つのエリアに分かれた新しいウィンドウが表示されます。

  • 段階的な変更。これは、ソースにコミットされる変更を表します。これはおそらく空白ですが、メインオプションの変更が表示されるかもしれませんので、それでも構いません。
  • 未段階の変更。Git にコミットすることを伝えていない、「保留」になっている変更点が表示されます。
  • コミットメッセージ。これは、変更が何であるかを説明するために追加できるメモです。

とりあえず、すべての変更をコミットしたいので、Stage Allボタンをクリックし、Commit Message に "First Commit!" などと入力し、Commit をクリックします。これで、すべてのリソースに緑のチェックマーク（ ）がつきました。これは、変更が見つからなかったことを意味します。

このウィンドウで作業する際、様々なStagedと Unstagedの メッセージの前に記号が表示されることに注意してください。使用される記号は操作やファイルによって異なり、次のようなものが使用されます。

  • [...] - ステージチェンジ（従来は[A]）。
  • [+] - 新たに追加されたもの
  • [->] - 名称変更・移動
  • [-] - 削除
  • [M] - アンステージド・チェンジ
  • [?] - 未トラック化ファイル
  • [!] - 欠落ファイル

これで私たちの作業プロセスは開始できます。上の段落で行ったのと同じように、プロジェクトを構築し、変更を保存してコミットすることができるのです。

しばらく作業をしていると、その時々にコミットした内容を確認したくなることがあります。そのような場合には、コミット履歴を開くとよいでしょう。履歴ウィンドウを開くには、Source Control>View Historyと進むと、3つのペインからなる別のウィンドウが表示されます。

上のペインにはコミット、中央のペインには選択したコミットに付けられたコメント、下のペインにはこのコミットで変更されたファイルが表示されます。Diff Tool を設定している場合は、このウィンドウ内の任意のファイルをダブルクリックすると diff ツールが起動し、ファイル間の変更点を見ることができることに注意してください。
