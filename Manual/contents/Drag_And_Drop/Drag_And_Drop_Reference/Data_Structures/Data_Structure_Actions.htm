データ構造アクションライブラリ

ゲームでは、情報を正確に順序立てて保存する必要があることがよくあります。例えば、人が持っているアイテムのリストを保存したり、まだ行く必要のある場所のグリッドを保存したりする必要があります。このような場合、複数の変数 しかし、データの並べ替えや特定の項目の検索など、より複雑な操作を行う場合は、実行速度が遅く、デバッグが困難で、書くのが本当に面倒なアクションブロックコードを大量に書かなければなりません。

そのため、GameMakerにはいくつかのデータ構造が組み込まれており、専門の関数を使ってアクセスすることができます。GML Visualで利用できるデータ構造は4種類あります（GMLを利用する場合は6種類）。 保存したい情報の種類や、後でどのように操作するかによって、それぞれ長所と短所があります - 4つのタイプがあります。

リスト リストは、情報を追加しながら順次格納していくデータ構造である。リストのインデックスは0から始まり、リストの末尾に項目が追加されるたびに上がっていきますが、末尾以外の位置にデータを挿入することも可能です。スタックデータ構造（後述）とは異なり、リストのどのインデックスからでも、いつでもデータにアクセスすることができます。
地図 Mapデータ構造は、非常に便利なデータ構造です。 キーと 値のペアです。例えば、RPGでヘルスポーションを10個、マナポーションを5個、ゴールドを100個持っているように、ゲームのキャラクターは様々なアイテム（キー）を大量に持ち、個々のアイテムに対してその個数（値）を持つことができます。マップはこのようなペアを一箇所にまとめて管理し、ペアをマップに追加したり、簡単な関数を使って特定のキーに対応する値を検索したりすることができます。マップは（認識できる）方法でソートされていません。つまり、あるキーを見つけるためには、全体を繰り返し見る必要があり（これは非常に時間がかかります）、また、同じキーを2つ保持する方法はありませんし、1つのキーに2つの値を割り当てることもできません。
グリッド グリッドは2次元の値の配列で、その幅と高さを定義することができます。これは、グリッド内のグリッド「セル」の総数（幅と高さの掛け算）を設定し、これらのセルは異なるデータ値を保持するために使用され、許可されたデータ型のいずれかにすることができます。この種の構造体では、グリッドの x (列) と y (行) の位置を与えることで、グリッド内のセルの値を設定したり取得したりできる (行と列はどちらも 0 から始まり、グリッド幅 - 1 とグリッド高さ - 1 まで)。グリッドのデータ構造にアクセスする場合、セル位置には常に整数値を使うようにする必要があり、非整数のインデックスはすべて GameMaker Studio 2の場合、そうでない 場合はもし、そうでない場合は、チェックしたいインデックスを渡す前に、あらかじめ自分で丸めをする必要があります。
スタック スタックデータ構造は、いわゆるLIFO（Last-in-First-out）構造である。スタックに値をプッシュし、スタックからポップすることで再び取り除くことができます。そして、最も最近スタックにプッシュされた値が、再びスタックからポップされる最初の値です（ちょうどコインのスタックを考えてください。）スタックは、割り込みを処理するときや、再帰的な関数を持つとき、あるいはゲームの初歩的なAIを構築するときにもよく使われます。

データ構造を作成し、そのインデックス値を変数に格納します。データ構造を作成し、そのインデックス値を変数に格納します。次に、このインデックスを使用して、データ構造に対して操作を行うことができるすべての関数呼び出しでデータ構造を参照します。最後に データ構造を破棄し、メモリから削除します。構造体は必要な数だけ同時に使用でき、すべての構造体がstrings と実数の両方の値を格納できます。

注：すべてのダイナミックリソースと同様に、データ構造はメモリを消費します。したがって、メモリリークによるゲームのスローダウンやクラッシュを防ぐために、不要になったら必ず破棄する必要があります。
