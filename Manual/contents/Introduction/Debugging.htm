デバッギング

GML Code、GML Visualを問わず、ゲームのプログラミングを行う際、間違った変数を使用したり、間違った引数を渡したり、間違った関数を使用したりと、非常に多くの間違いを犯す可能性があります。つまり、これらの間違いは、実際にゲームを実行したり、コンパイルしたときに初めて明らかになり、そのときでさえ、何が間違っているのかを正確に知ることは困難な場合があるのです。そのため、選択したデバイスでゲームがどのように動作しているか、room にどのようなインスタンスがあるか、どのようなグローバル変数が作成されているか、その他多くの詳細を確認することが非常に重要です。これらのことを確認することを「コードのデバッグ」と呼び、見つけたエラーや問題を「バグ」と呼びます。

このようなことは、いわゆる「デバッグコード」をゲームに追加して必要な情報を画面に表示させたり、エラー時にメッセージを表示させたりすることで、自分で確認することができますが、手間がかかることが多く、そもそもどこに問題があるのかがわからない場合は、バグを突き止めるために試行錯誤することになり、時間がかかる場合があります。その点、GameMakerには Debug Modeという特別なテストモードがあり、Debug Moduleが起動します。また、Compiling時にメッセージを表示し、ゲームの問題を追跡するための包括的なエラー報告システムも用意されています。

以下のセクションでは、ゲームをデバッグするためのさまざまな方法について説明します。

エラー報告

プロジェクトのコンパイルや実行中にエラーが発生した場合、「コンパイルエラー出力ウィンドウ」に以下のようなメッセージが表示されます。

ほとんどのコンパイラーエラーメッセージは、エラーが発生したobject を示すメッセージ、次にイベントと行番号、そして最後にエラー自体の短い説明と、同じアウトラインをたどっています。エラーレポートによって提供されるこの種の情報は、完成したゲームがクラッシュする原因となる構文や言語関連のバグを検出して報告するため、非常に重要です。そのような場合は、デバッグモード（以下で説明）を使用する必要があります。

これらのコンパイルエラーとは別に、シンタックスエラーが発生することもあります。これらはGML Visual アクションやコードを追加する際に検出され、下図に示すようなシンタックス・エラー出力ウィンドウに表示されます。

コンパイラーエラーと同様に、シンタックスエラーウィンドウは、object 、イベントと問題を与えている特定の行を指示し、多くの場合、エラーの内容を簡単に説明するために表示されています。

デバッグモード

より慎重に確認したい場合は、デバッグ実行ボタン（ ）をクリックし て、デバッグモードでゲームを実行することができます。そうすると、デバッグモジュールがゲームと一緒に開かれます。このモジュールでは、グローバルなパフォーマンスから、個々のインスタンスの個々の変数の値に至るまで、ゲーム内のすべての動作を監視することができます。Debug Module の詳細とその使い方については、次のページを参照してください。

  • デバッガ

Debug Module を使用する場合、Breakpoints を コードまたはGML Visual に追加することもできることに注意してください。breakpoint は、ゲームloop で、デバッグモジュールを一時停止して、実際のゲームコードを一度に 1 行ずつステップスルーさせたいポイントです。F9」キーでbreakpoints を追加できます。これらはOutput Windowに表示され、必要に応じて有効/無効を切り替えることができます。

追加ツール

debugging コンパイル済みゲームの追加ツールとして、最終的なゲームパッケージをWindows で実行する際に使用できるコマンドラインパラメータが存在します。この仕組みについては、ここを参照してください。

また、ゲームのテスト実行時にデバッグオーバーレイを生成して、実行中のゲームのパフォーマンスに関する追加のデバッグ情報を表示することもできます。さらに、コンパイルコンソールにデバッグメッセージを出力して、ゲームの動作のデバッグに役立つ独自のテキストを提供することも可能です。詳しくは以下のページをご覧ください。

  • show_debug_overlay
  • show_debug_message
  • debug_event
