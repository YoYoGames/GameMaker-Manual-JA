Feather は、GML コード scripts において、インテリジェントなコード補完と高度な構文チェックを提供し、スマートなリファクタリングオプションも提供します。

このページでは、「Feather」の以下の部分について情報を掲載しています。

  • ベースオプション
  • ハイライト表示
  • メッセージの重要度(「GMXXXX」ルール)
  • 命名規則(assets, 変数などの場合)

ベースオプション

ベースとなるFeather Settingsには、以下のオプションがあります。

  • Featherを有効にする。ここで Feather を有効化または無効化します。無効にすると、IDE 全体で Feather の機能は利用できなくなります。
  • Document Parse Delay (ドキュメント解析遅延)。script に変更を加えた後、Feather がチェックするために待つ時間です。
  • コンパクトなツールチップ。この機能を有効にすると、関数のパラメータ説明など、キーワードにカーソルを合わせたときに表示される特定のツールチップ情報を削除し、ツールチップを小さくすることができます。
  • ツールチップで表示されるstruct フィールドの最大数です。struct の参照にカーソルを合わせると、そのstruct 内の変数がツールチップで表示されます（ここに入力した数まで）。
  • asset がasset ブラウザーでリネームされたときにAsset の名前への参照をリファクタリングします。これを有効にすると、asset ブラウザでAsset の名前を変更したときに、コード内のその名前に関するすべての言及も編集され、それらの参照が壊れないようにします。
  • Quick Fixから作成する場合、複数行のJSDocコメント・スタイルを優先するようにしました。QuickFixesメニューから関数のJSDocコメントを生成する場合、このオプションによって、生成されるJSDocは単一行または複数行のコメントを使用するようになります。
  • Strict Type モードを有効にします。このモードを有効にすると、コード内で正しいデータ型が使用されているかどうかを確認するために、より厳格になります。

ハイライト表示

このセクションでは、エラー、警告、提案を示すためにコードの一部に下線を引く際に使用する色を変更するための設定について説明します。

  • エラー色：エラーに使用される色。
  • 警告色：警告に使用する色。
  • サジェスチョンカラー：サジェスチョンに使用される色です。

メッセージの重要度

このセクションには、Feather がコードのチェックに使用するすべてのルールが含まれており、各ルールの重大度を調整することができます。

プロフィール

このドロップダウンボックスでは、GM*ルールのプリセットをいくつか選択することができます。以下のプロファイルが利用可能です。

  • なし。このプロファイルは、エラーメッセージを表示しません。
  • SyntaxErrors。このプロファイルを使用すると、Noneプロファイルで表示される内容に加えて、シンタックスエラーが表示されます。
  • TypeErrors（タイプエラー）。このプロファイルは、SyntaxErrors プロファイルに加え、タイプエラーも表示します。
  • すべて：Featherが解析できるすべてのものを表示するプロファイルです。

GM*ルール

ルールを「エラー」「警告」「提案」に設定することで、コードエディタやFeatherメッセージウィンドウでルール違反のコードが報告される方法に影響を与えることができます。

ルールを「無視」に設定すると、そのルールに違反したすべてのインスタンスを無視することができます。

GM1000」から始まるルールは、致命的なエラーがないかをチェックするもので、「GM2000」から始まるルールは、ゲームのバグを防ぐためのベストプラクティス（最善の方法）ルールです。

ネーミングルール

このセクションでは、assets 、変数、およびGML コードの様々な部分の命名規則を設定することができます。

ネーミングルールを有効にするには、「メッセージの重要度」の GM2017 ルールが有効である必要があります。デフォルトでは、無効になっています。

このセクションの設定は次のとおりです。

  • Identifiier Blocklist (識別子ブロックリスト)。すべての命名規則で無視される識別子をスペースで区切ったリストです。

    たとえば、objects のプレフィックスとしてobj_を使用するルールがあるが、objects マネージャと ネットワークはそのままにして、obj_プレフィックスを使用しないようにしたいとします。

    この場合、manager networkをIndentifier Blocklistフィールドに記述します。Feather はこれらのobjects と、同じ名前の他の識別子（変数、列挙型、パラメータなど）を無視します。
  • <identifier> 命名規則:assets, マクロ, enum, 関数パラメーター, 変数など、リストにある識別子の種類ごとに命名規則を設定することができます。

    各命名ルールのドロップダウンには、以下のオプションがあります。
    • プレフィックス。objects の場合はobj_、ローカル変数の場合は_といった具合に、好みに応じて指定します。
    • サフィックス。識別子名の後につけるテキスト。
    • ネーミング・スタイル識別子のネーミング・スタイルを選択します。オプションは以下の通りです。
      • 制約なし。このタイプの識別子には、何のフォーマットも強制されません。
      • UpperCamelCase。MyHealth、ObjGrappleHook、MaxInventoryItemsAllowed、など。
      • lowerCamelCase:myHealth, objGrappleHook, maxInventoryItemsAllowed,etc.
      • ALL_UPPER:MY_HEALTH, OBJ_GRAPPLE_HOOK, MAX_INVENTORY_ITEMS_ALLOWED、など。
      • ALL_LOWER:MY_HEALTH、OBJ_GRAPPLE_HOOK、MAX_INVENTORY_ITEMS_ALLOWED、etc.
